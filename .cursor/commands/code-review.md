# /code-review — 実装差分のコードレビュー

目的: 変更差分を精査し、重大度ごとに指摘・テスト不足・残リスクを整理して報告する。

手順:
1. 差分の取得
   - `git status -sb` で変更ファイルを確認。
   - `git diff --stat` と `git diff`（必要に応じてファイルごと）で内容を把握。
2. 重大度判定
   - Critical: クラッシュ/データ破壊/セキュリティ欠陥。
   - High: 仕様逸脱・リリースブロッカー。
   - Medium: UX低下・メンテ性低下・パフォーマンス懸念。
   - Low: 軽微な改善提案/スタイル。
   - 指摘ごとに「重大度 / 概要 / 根拠（`path:line-line` + サマリ） / 影響 / 推奨アクション」を明記。
3. テスト観点
   - 既存テストの影響や不足テストを列挙（種別: unit/integration/e2e 等）。
   - 必要なら具体的なテストケース案を提示。
4. ポジティブフィードバック（任意）
   - 良い実装や改善点を短く記載し、モチベーション維持を図る。
5. レポート構成
   - `Findings (重大度順)` -> 箇条書き。
   - `Testing` -> 実施済み/不足/推奨。
   - `Residual Risks` -> 解消されていない懸念。
   - `Notes` -> ポジティブコメントや次のアクション。

備考:
- 指摘が無い場合も「No findings」と明示し、残リスクや今後の注意点を記載する。
- 修正を提案する際は、ユーザーの許可を得てから実装に入る。
